\chapter{Documentazione e OpenAPI}

La documentazione è fondamentale per l'adozione e l'utilizzo corretto di un'API. OpenAPI Specification (precedentemente Swagger) è lo standard de facto per documentare REST API.

\section{Introduzione a OpenAPI}

\subsection{Cos'è OpenAPI}

OpenAPI Specification (OAS) è uno standard per descrivere REST API in formato machine-readable (YAML o JSON).

\begin{tcolorbox}[title=Benefici di OpenAPI]
\begin{itemize}
\item \textbf{Documentazione interattiva}: Genera Swagger UI automaticamente
\item \textbf{Code generation}: Genera client/server in vari linguaggi
\item \textbf{Validazione}: Valida request/response automaticamente
\item \textbf{Testing}: Genera test cases automaticamente
\item \textbf{Mocking}: Crea mock server per sviluppo
\item \textbf{Contract-first}: Progetta API prima di implementare
\end{itemize}
\end{tcolorbox}

\subsection{Versioni OpenAPI}

\begin{itemize}
\item \textbf{OpenAPI 2.0} (Swagger 2.0): Versione legacy
\item \textbf{OpenAPI 3.0}: Versione attuale (focus di questo capitolo)
\item \textbf{OpenAPI 3.1}: Versione più recente, compatibile con JSON Schema
\end{itemize}

\subsection{Documento Minimale}

                items:

\subsection{Sezioni Principali}

\begin{enumerate}
\item \textbf{openapi}: Versione della spec
\item \textbf{info}: Metadati dell'API
\item \textbf{servers}: URL dei server
\item \textbf{paths}: Endpoint e operazioni
\item \textbf{components}: Componenti riutilizzabili (schemas, responses, etc)
\item \textbf{security}: Schemi di sicurezza
\item \textbf{tags}: Raggruppamento endpoint
\item \textbf{externalDocs}: Documentazione esterna
\end{enumerate}

app = Flask(__name__)

@app.route('/api/openapi.yaml')

@app.route('/api/openapi.json')
def openapi_spec_json():
    """Serve OpenAPI as JSON"""
    with open('openapi.yaml') as f:

\subsection{Validare Request/Response}

\begin{lstlisting}[language=Python, caption=Validazione Runtime con OpenAPI]
from flask import Flask, request, jsonify
from openapi_core import create_spec
from openapi_core.validation.request.validators import RequestValidator
from openapi_core.contrib.flask import FlaskOpenAPIRequest

app = Flask(__name__)

# Carica spec
with open('openapi.yaml') as f:
    spec_dict = yaml.safe_load(f)
spec = create_spec(spec_dict)

# Middleware validazione
@app.before_request
def validate_request():
    """Valida request contro OpenAPI spec"""
    if request.method in ['POST', 'PUT', 'PATCH']:
        openapi_request = FlaskOpenAPIRequest(request)
        validator = RequestValidator(spec)
        result = validator.validate(openapi_request)

        if result.errors:
            errors = [
                {
                    'path': '.'.join(str(p) for p in err.schema_path),
                    'message': err.message
                }
                for err in result.errors
            ]

            return jsonify({
                'error': {
                    'code': 'schema_validation_error',
                    'message': 'Request does not match OpenAPI schema',
                    'errors': errors
                }
            }), 400

\section{Best Practices}

\begin{tcolorbox}[title=Checklist OpenAPI]
\begin{itemize}
\item[$\square$] Usa versione OpenAPI 3.0 o superiore
\item[$\square$] Includi esempi per tutti gli schema
\item[$\square$] Documenta tutti i parametri e proprietà
\item[$\square$] Definisci tutti i possibili status code
\item[$\square$] Usa \$ref per riutilizzare componenti
\item[$\square$] Testa la spec con validator
\item[$\square$] Mantieni spec sincronizzata con codice
\item[$\square$] Versionna la spec insieme al codice
\item[$\square$] Genera e testa i client
\item[$\square$] Usa descrizioni Markdown per formattazione
\end{itemize}
\end{tcolorbox}

\section{Riepilogo}

OpenAPI 3.0 è lo standard per documentare REST API. Fornisce:

\begin{itemize}
\item Documentazione interattiva con Swagger UI
\item Validazione automatica request/response
\item Code generation per client e server
\item Contract-first development
\item Testing automatizzato
\end{itemize}

Investire tempo in una spec OpenAPI completa ripaga enormemente in termini di developer experience e manutenibilità.
